## Hacker Perspective in React XSS

| حمله                        | توضیح                                                        | ابزار یا تکنیک                         |
| --------------------------- | ------------------------------------------------------------ | -------------------------------------- |
| **Reflected XSS**           | تزریق ورودی کاربر در DOM بدون sanitation                     | پارامترهای URL، کوئری‌ها                |
| **DOM-based XSS**           | وقتی JS خودش از `location`, `document.write`, `eval`, یا innerHTML استفاده می‌کنه | ابزار: DOM Invader, DevTools           |
| **dangerouslySetInnerHTML** | بزرگ‌ترین در پشتی برای اجرای HTML/JS دلخواه در DOM            | تزریق مستقیم HTML به اجزای صفحه        |
| **Third-party libs**        | استفاده از پکیج‌هایی مثل `marked`, `dompurify` بدون تنظیمات امن | بررسی sourcecode و misconfig           |
| **JSX misusage**            | استفاده اشتباه از binding‌ها یا نمایش مستقیم داده‌ها           | مثل `{data}` بدون validation یا encode |

  return <div dangerouslySetInnerHTML={{ __html: bio }} />;
}

## 🧠 جاهایی که باید شک کنیم (در Source Audit یا BlackBox):

1. `dangerouslySetInnerHTML`
2. `eval`, `Function`, `setTimeout("code", ...)`
3. `ref`های custom به عناصر DOM
4. componentهایی که `input.value`, `props.children`, `html`، یا `innerText` بدون validation نمایش می‌دن
5. داده‌هایی که از API برگشتن و مستقیم وارد DOM می‌شن

## 📍 نکات طلایی

- React خودش به‌طور پیش‌فرض JSX رو Escape می‌کنه، ولی فقط **در تگ‌ها**. اگه جایی از `dangerouslySetInnerHTML` یا `iframe.contentWindow.eval` استفاده شد → شکارگاه توئه.
- داده‌هایی که مستقیم از backend میان و در کامپوننت‌هایی مثل `Modal`, `Toast`, `Card` نمایش داده می‌شن، بسیار آسیب‌پذیرن اگر encode نشده باشن.

در این مقاله، آسیب‌پذیری XSS در کتابخانه‌های شخص ثالث مورد استفاده در برنامه‌های React بررسی شده است. مثال‌هایی از نحوه بهره‌برداری از این آسیب‌پذیری‌ها و روش‌های پیشگیری ارائه شده است.

## 🛠️ ابزارهای مفید برای شناسایی و بهره‌برداری از XSS در React

- **DOM Invader (Burp Suite)**: ابزاری برای شناسایی sink‌های خطرناک در صفحات React.
- **Dalfox**: اسکنر خودکار XSS با قابلیت شناسایی آسیب‌پذیری‌ها حتی در اپلیکیشن‌های SPA.
- **XSStrike**: ابزاری برای تست context-aware و یافتن payload مناسب.
- **React DevTools + Watch DOM**: ابزاری برای بررسی binding‌های live در DOM.
- **JS Parser مثل LinkFinder**: ابزاری برای شناسایی sink و source‌های خطرناک.

## 🔎 ۳. مراحل تحلیل یک فایل JS مشکوک (Obfuscated)

1. **Pretty Print در DevTools**
    اگر سورس مین‌فای شده است، اول با دکمه `{}` در DevTools کد رو readable کن.

2. **جستجوی عبارت‌های مشکوک**
    دنبال `eval`, `Function`, `setTimeout("code")`, `document.write`, `atob`, `window[...]`, `String.fromCharCode`, یا توابع با نام‌های بی‌معنی بگرد.

3. **استفاده از Breakpoint در زمان اجرا**
    روی `eval` یا `script tag`ها breakpoint بذار. ببین کجا کد decypher یا اجرا می‌شه.

4. **جایگزینی `eval` با `console.log`**
    اگر کدی شبیه به این دیدی:

   ```js
   eval(function(p,a,c,k,e,d)...)
   ```

   می‌تونی `eval` رو موقتاً با `console.log` جایگزین کنی تا payload اصلی رو ببینی.

5. **Decode کردن دستی یا با ابزار**
    اگر stringهای مبهم داری مثل:

   ```js
   var _0xabc=["\x61\x6C\x65\x72\x74"];
   ```

   از ابزارهایی مثل **de4js** یا **CyberChef** استفاده کن.

## 🧪 مثال عملی

```js
jsCopyEditvar _0x5fa3=["\x61\x6C\x65\x72\x74","\x68\x65\x6C\x6C\x6F"];
window[_0x5fa3[0]](_0x5fa3[1])
```

در بالا `alert("hello")` داره اجرا می‌شه. تو باید بتونی:

1. مقدارهای `_0x5fa3` رو بفهمی
2. بگردی ببینی چه متدی (در اینجا `alert`) اجرا می‌شه
3. stringها رو decode کنی
